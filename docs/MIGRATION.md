# Guardrails migration guide

These migrations remove compatibility paths that could make an old setup look
protected while bypassing the current trust model. Migrate the repository and
record fresh evidence; do not copy old generated artifacts forward.

## Agentcam 0.4 → 0.5

- Install Agentcam 0.5 in the repository runtime and start a new turn/session.
- Re-run verification against the final product state. Manifests without the
  turn delta and state fingerprints are `legacy-unbound`, not `local-recorded`.
- Replace `[recorded by agentcam]` with the handoff generated by 0.5, which uses
  the exact `[locally recorded by agentcam]` marker.
- Hook mode has no terminal streams. It now reports stdout/stderr as unavailable
  and creates no empty log placeholders.

## Corridor CI v12 → v13

- Pin the workflow to `corridor-ci-v13.0.0`.
- Use only the five-line handoff grammar and the exact Agentcam 0.5 marker.
- Use `Risk: high`, `Risk: medium`, `Risk: none-detected`, or `Risk: unknown`.
- Dependency changes need an OWNER/MEMBER comment bound to the current head SHA.
  Workflow changes in this repository need the corresponding Policy Gate
  approval bound to that same SHA.

Old recorded-looking markers and evidence whose product fingerprint differs
from the current PR are `unverified`.

## inline command → trusted check

Corridor Markdown never authorizes a shell command. Register argv in the trusted
Git-local configuration, then refer to its ID:

```bash
guardrails check set primary -- python -m pytest -q
```

```md
## Stop Condition
- Check: primary
```

For `high` rigor, register a second ID and use `Independent check: integration`.
Its ID and normalized argv must differ from every primary check. The label means
a secondary trusted check, not proof of organizational independence.

`SLIME_TEST_CMD` and `SLIME_TYPECHECK_CMD` are ignored. Delete them from local
automation after registering trusted checks.

## corridor without Rigor → explicit tier

Every active corridor must contain exactly one explicit tier:

```md
## Rigor
trivial
```

Choose `normal` when evidence and falsification are needed, and `high` when the
failure mode requires rollback plus a distinct secondary check. A corridor with
no Rigor is rejected; it is not treated as a legacy fallback.
